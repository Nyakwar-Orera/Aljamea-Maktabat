### 🏛️ Multi-Campus Architecture Proposal

#### 1. **Branch-Aware Authentication & Session Management**
*   **User Mapping:** Extend the `appdata.db` user table to include a `branch_id` or `branch_code`.
*   **Intelligent Redirection:** Upon login, the system detects the user's branch and automatically redirects them to `/dashboard/<branch_code>`.
*   **Session Isolation:** Store the `current_branch` in the secure Flask session to ensure that no data from private campuses leaks into another.

#### 2. **Dynamic Koha Query Injection**
Currently, queries extract data globally. To filter by branch:
*   **Koha Branch Codes:** Koha stores library branches in the `branches` table with unique shortcodes. 
*   **Filtered SQL Logic:** Update the `koha_queries.py` service to accept a `branch_filter` parameter in every function. 
    *   *Example:* Filtering `statistics` and `items` by the `homebranch` column to ensure an Admin only sees books and circulation data belonging to their specific campus.
*   **Centralized Middleware:** Create a "Branch Interceptor" that adds a `WHERE homebranch = ?` clause to all queries before execution.

#### 3. **The Super Admin "Global Command Center"**
*   **Unified Dashboard:** A top-level view for Institutional Leaders (Super Admins) that aggregates data from all four campuses into a single screen.
*   **Branch Selector:** A sleek dropdown in the navbar for Super Admins to "hot-swap" their view between Nairobi, Surrey, etc., without logging out.
*   **Cross-Campus Comparison:** Visualizations that compare borrowing trends, library program attendance, and **Taqeem** scores across all branches to identify institutional strengths.

#### 4. **Scalable Data Structure**
*   **Shared App Metadata:** Maintain a single `appdata.db` for the portal logic while seamlessly connecting to the shared (or individual) Koha MySQL backends.
*   **Branch-Specific Reporting:** Automated monthly reports ([tasks/monthly_reports.py](cci:7://file:///home/nyakwarorera/Desktop/Aljamea-Maktabat/tasks/monthly_reports.py:0:0-0:0)) would be triggered separately for each campus HOD, ensuring they receive only their local data.
