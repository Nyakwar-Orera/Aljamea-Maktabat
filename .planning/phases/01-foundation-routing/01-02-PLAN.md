# Plan: Parallel Query Engine

This plan implements high-performance multi-campus data retrieval using Python's `ThreadPoolExecutor`.

## Tasks

<task>
<read_first>
- `db_koha.py`
- `config.py`
</read_first>
<action>
Refactor `db_koha.py` to support multi-host connection pooling.
Ensure that each branch in `CAMPUS_REGISTRY` can have its own `MySQLConnectionPool`.
Implement a `get_db_connection(branch_code)` helper that retrieves one connection from the correct pool.
</action>
<acceptance_criteria>
- `db_koha.py` supports multiple pools.
- Connections are returned to the pool even on errors.
</acceptance_criteria>
</task>

<task>
<read_first>
- `services/koha_queries.py`
- `services/branch_queries.py`
</read_first>
<action>
Implement `GlobalQueryEngine` in `services/branch_queries.py`.
The engine should:
1. Initialize a `ThreadPoolExecutor`.
2. Provide a `fetch_all_campuses(query_fn)` method.
3. Use `executor.submit` to run queries across all active campuses in parallel.
4. Collect results and handle timeouts/errors gracefully (return empty results for a branch if it fails).
</action>
<acceptance_criteria>
- `services/branch_queries.py` contains `GlobalQueryEngine`.
- Engine uses `concurrent.futures.ThreadPoolExecutor`.
- Engine respects the `active: True` flag in `CAMPUS_REGISTRY`.
- Simultaneous connections are established during execution.
</acceptance_criteria>
</task>

<task>
<read_first>
- `routes/super_admin.py`
</read_first>
<action>
Implement a test route `/super_admin/check_parallel` that fetches total branch counts from all campuses using the new engine.
Render results on a simple test page.
</action>
<acceptance_criteria>
- `/super_admin/check_parallel` successfully displays counts from all active branches.
- Page load time is significantly less than the sum of sequential query times.
</acceptance_criteria>
</task>

## Verification Criteria

- [ ] Parallel engine returns data for all active branches in a single request.
- [ ] Timeouts are handled without crashing the global dashboard.
- [ ] Database connections stay within pool limits even under rapid refresh.

## must_haves

- `concurrent.futures` implementation.
- Branch-isolated connection pooling.
